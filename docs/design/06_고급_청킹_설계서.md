# 고급 텍스트 청킹 설계서

> 기존 고정 크기 청킹(`chunker.py`)을 대체할 고급 청킹 전략 설계.
> 구현 파일: `app/services/chunker_v2.py` (미연결 상태, 검토 후 교체 가능)

---

## 1. 현황 및 문제점

### 기존 방식 (`app/services/chunker.py`)
```python
# 고정 크기 문자 분할 — 문장/단어 경계 무시
while start < len(text):
    chunks.append(text[start:start + chunk_size])
    start += chunk_size - chunk_overlap
```

**문제점**:
- 단어/문장 중간에서 잘림 (테스트 결과 7개 청크 중 **6건 문장 절단**)
- 의미 단위 무시 — 한 주제가 두 청크에 걸침
- 논문 섹션 구조 무시 — Introduction과 Methods가 하나의 청크에 섞임
- RAG 검색 시 불완전한 컨텍스트 제공

## 2. 최신 청킹 기술 조사 결과 (2024~2026)

### 2.1 Recursive Chunking
- **출처**: LangChain RecursiveCharacterTextSplitter, NVIDIA 2024 벤치마크
- **원리**: 구분자 우선순위(`\n\n` > `\n` > `. ` > ` ` > `""`)로 재귀 분할
- **성능**: NVIDIA 권장 400~512 토큰, 10~20% overlap
- **비용**: 없음 (순수 문자열 처리)

### 2.2 Semantic Chunking
- **출처**: LangChain experimental, Chonkie, Milvus 블로그 (2025)
- **원리**: 문장별 임베딩 → 인접 문장 코사인 유사도 비교 → 유사도 급락 지점에서 분할
- **성능**: recall 0.913~0.919 (Max-Min Semantic Chunking)
- **비용**: 모든 문장 임베딩 필요
- **주의**: NAACL 2025 논문 — 고정 200단어 대비 일관된 우위 없음 (쿼리 유형에 따라 다름)

### 2.3 Late Chunking (Jina AI, 2024)
- **출처**: arXiv:2409.04701, Jina AI
- **원리**: "임베딩 → 청킹" 순서 (전체 문서 임베딩 후 청크 경계에서 mean pooling)
- **장점**: 청크가 문서 전체 맥락 보존, 대명사/참조 해결
- **단점**: Long-context 임베딩 모델 필요 (jina-embeddings-v3), 폐쇄망 제한

### 2.4 Agentic Chunking (LLM 기반)
- **출처**: IBM 2024, LangChain + LLM
- **원리**: LLM이 각 명제(proposition)가 기존 청크에 속하는지 판단
- **성능**: 이론적 최고 품질
- **단점**: 극도로 높은 LLM 비용, NAACL 2025 "계산 오버헤드로 실험 중단"
- **전망**: LLM 속도/비용 개선 시 실용화 (2026~)

### 2.5 Contextual Retrieval (Anthropic, 2024)
- **출처**: Anthropic 공식 블로그
- **원리**: 일반 청킹 후 각 청크에 LLM으로 문서 맥락 요약 prepend
- **성능**: 검색 오류 **49% 감소** (+ reranking 시 67%)
- **비용**: 100만 토큰당 약 $1.02 (일회성 인덱싱)

### 2.6 Section-Aware Chunking (논문 특화)
- **출처**: MultiDocFusion (EMNLP 2025), TreeRAG (ACL Findings 2025)
- **원리**: 논문 섹션 구조(Abstract, Introduction, Methods...)를 감지하여 섹션 경계 보존
- **장점**: 섹션 메타데이터 포함, 논문에 최적
- **비용**: 없음 (regex 기반 섹션 감지)

## 4. 구현: chunker_v2.py

### 4.1 5가지 전략

`app/services/chunker_v2.py`에 아래 5가지 전략을 구현:

| # | 전략 | 설명 | 기본값 |
|---|------|------|--------|
| 1 | `fixed` | 기존과 동일 (호환용) | |
| 2 | `recursive` | 구분자 우선순위 재귀 분할 | ★ 기본값 |
| 3 | `sentence` | 문장 단위 분할 후 병합 | |
| 4 | `semantic` | 임베딩 유사도 기반 의미 경계 (async) | |
| 5 | `section` | 논문 섹션 구조 인식 + recursive | ★★ 논문 추천 |

### 4.2 인터페이스

```python
# drop-in replacement (기존과 동일 시그니처)
def chunk_text(text, chunk_size=None, chunk_overlap=None) -> list[str]

# 전략 지정
def chunk_text_with_strategy(text, strategy="recursive", chunk_size=None, chunk_overlap=None) -> list[str]

# 비동기 (semantic 전략에서 임베딩 호출 필요 시)
async def chunk_text_async(text, strategy="semantic", chunk_size=None, chunk_overlap=None) -> list[str]
```

### 4.3 Recursive 전략 상세

```
구분자 우선순위:
  "\n\n" (단락) → "\n" (줄) → ". " (문장) → "? " → "! " → "; " → ", " → " " (공백) → "" (문자)

처리 흐름:
  1. 최상위 구분자로 분할
  2. 조각이 chunk_size 이하면 → 다음 조각과 병합 (chunk_size 이내로)
  3. 조각이 chunk_size 초과면 → 다음 레벨 구분자로 재귀 분할
  4. overlap 처리: flush 시 마지막 조각들을 overlap 크기만큼 유지
```

### 4.4 Section 전략 상세

```
처리 흐름:
  1. regex로 논문 섹션 감지:
     - 대문자: ABSTRACT, INTRODUCTION, METHODS, RESULTS, ...
     - 번호: "1. Introduction", "2.1 Finite Element Method Setup"
     - Title Case: Introduction, Related Work, ...
  2. 섹션별로 텍스트 분리
  3. 각 섹션을 recursive chunking으로 분할
  4. 각 청크 앞에 "[Section: 섹션명]" 메타데이터 추가
  5. 섹션 미감지 시 → 전체를 recursive로 fallback
```

### 4.5 Semantic 전략 상세

```
처리 흐름:
  1. 텍스트를 문장으로 분할
  2. 모든 문장을 배치 임베딩 (embed_texts)
  3. 인접 문장 간 코사인 유사도 계산
  4. percentile 기반 임계값 결정 (기본: 하위 10%)
  5. 유사도 < 임계값인 지점에서 분할
  6. 큰 그룹은 sentence chunking으로 재분할

파라미터:
  - similarity_threshold: 최소 유사도 (기본 0.5)
  - breakpoint_percentile: 분할 기준 백분위수 (기본 90)
```

## 5. 벤치마크 결과

논문 텍스트(~3,000자)로 테스트 (chunk_size=512, overlap=50):

| 전략 | 청크 수 | 평균 크기 | 문장 절단 | 특징 |
|------|---------|----------|----------|------|
| fixed (현재) | 7 | 475자 | **6건** | 단어/문장 중간 절단 |
| recursive | 10 | 302자 | 2건 | 단락 경계 존중 |
| sentence | 7 | 432자 | 1건 | 문장 경계 보존 |
| **section** | **8** | **388자** | **0건** | 섹션 메타데이터 포함, 절단 없음 |

**핵심 개선**: section 전략은 문장 절단 0건 + 섹션 컨텍스트 보존.

## 6. 교체 가이드

### 6.1 즉시 교체 (1줄 변경)

`app/services/chunker.py`의 내용을 아래로 교체:

```python
"""Text chunker — chunker_v2 위임."""
from app.services.chunker_v2 import chunk_text  # noqa: F401
```

또는 `load_mariadb_to_milvus.py` 등에서 import 변경:

```python
# Before
from app.services.chunker import chunk_text

# After
from app.services.chunker_v2 import chunk_text
```

### 6.2 전략 변경

`.env`에 추가:
```env
CHUNKING_STRATEGY=section   # fixed | recursive | sentence | semantic | section
```

`app/config.py`에 추가:
```python
CHUNKING_STRATEGY = os.getenv("CHUNKING_STRATEGY", "recursive")
```

### 6.3 Milvus 재적재

청킹 전략 변경 후에는 Milvus 데이터 재적재 필요:

```bash
# 1. Milvus 컬렉션 삭제 (또는 새 컬렉션명 사용)
# 2. 재적재
python load_mariadb_to_milvus.py
```

### 6.4 단계별 도입 추천

1. **1단계**: `recursive` 전략으로 교체 (비용 0, 즉시 개선)
2. **2단계**: `section` 전략 테스트 (논문 구조 활용)
3. **3단계**: `semantic` 전략 A/B 테스트 (임베딩 비용 발생)
4. **(선택)**: Contextual Retrieval 추가 (LLM 비용, 별도 구현 필요)
