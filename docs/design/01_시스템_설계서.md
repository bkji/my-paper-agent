# Co-Scientist Agent 시스템 설계서

## 1. 프로젝트 개요

### 1.1 목적
논문 데이터(CSV)를 MariaDB(관계형)와 Milvus(벡터)에 적재하고, 14개 전문 에이전트가 사용자의 자연어 질문에 답하는 R&D 지원 시스템.

### 1.2 입력 데이터
- `data/sample_paper.csv` — 논문 원본 데이터 (17개 컬럼)
- 컬럼: `mariadb_id, filename, doi, coverdate(INT64 YYYYMMDD), title, paper_keyword, paper_text, volume, issue, totalpage, referencetotal, author, references, chunk_id, chunk_total_counts, bm25_keywords, parser_ver`

### 1.3 기술 스택

| 구분 | 기술 | 역할 |
|------|------|------|
| 웹 프레임워크 | FastAPI | REST API 서버 |
| 에이전트 프레임워크 | LangGraph | 상태 머신 기반 에이전트 |
| 관계형 DB | MariaDB | 논문 원본 저장, SQL 집계 |
| 벡터 DB | Milvus | 의미 검색 (Dense + Sparse) |
| LLM | OpenAI-compatible API | 의도 분류, 답변 생성 |
| 임베딩 | OpenAI-compatible API | bge-m3 (1024차원) |
| 트레이싱 | Langfuse | LLM 호출 추적, 비용 분석 |
| ORM | SQLAlchemy | MariaDB 접근 |
| 스트리밍 | SSE (Server-Sent Events) | 실시간 토큰 스트리밍 |

### 1.4 핵심 설계 원칙
1. **OpenAI-compatible API 통일** — LLM, Embedding 모두 OpenAI 포맷. `.env`만 변경하면 폐쇄망 이전 가능.
2. **Milvus=의미검색, MariaDB=집계** — 용도별 DB 분리.
3. **Milvus chunk 주의** — 논문이 chunk 단위로 적재됨. 편수 카운트는 `chunk_id=1`만.
4. **환경변수 일원화** — 모든 접속정보는 `.env`에서 관리. 코드 하드코딩 금지.

## 5. 에이전트 아키텍처

### 5.1 Supervisor 3단계 파이프라인

```
입력: {query, messages, user_id, filters}
  │
  ▼
[Stage 1] build_history
  - messages 배열 → conversation_history 문자열
  - 최근 5턴(10항목) 유지
  │
  ▼
[Stage 2] extract_dates + extract_conditions
  - regex 기반 날짜 파싱 (D1~D4)
  - DOI 추출 (10.XXXX/YYYY 패턴)
  - LLM 기반 키워드/저자/volume/issue 추출
  - 결과 → state.filters
  │
  ▼
[Stage 3] classify_intent
  - LLM이 사용자 질문을 14개 에이전트 유형 중 하나로 분류
  - 분류 기준:
    - analytics: 통계, 편수, 목록, 추이
    - paper_qa: 논문 검색, 일반 질문
    - literature_survey: 리뷰, 서베이, 동향 파악
    - paper_deep_dive: 특정 논문 심층 분석
    - 등등...
  │
  ▼
[Stage 4] route_to_agent
  - agent_type에 따라 해당 에이전트 실행
  │
  ▼
[Stage 5] append_citation
  - sources 기반 참조 문헌 포맷 추가
  - 저작권 고지 추가
  │
  ▼
출력: {answer, sources, trace_id, usage}
```

### 5.2 14개 에이전트 목록

| Phase | 에이전트 | 역할 | 데이터소스 |
|-------|---------|------|-----------|

| 1 | literature_survey | 문헌 리뷰 생성 | Milvus |

| 1 | analytics | 통계/집계/목록 | MariaDB (SQL) |
| 2 | idea_generator | 연구 아이디어 생성 | Milvus |
| 2 | cross_domain | 타분야 적용 아이디어 | Milvus |
| 2 | trend_analyzer | 기술 트렌드 분석 | Milvus |
| 3 | experiment_planner | 실험 설계 제안 | Milvus |
| 3 | material_advisor | 재료/공정 분석 | Milvus |
| 3 | patent_landscaper | 특허 동향 분석 | Milvus |
| 3 | competitive_intel | 경쟁사 모니터링 | Milvus |
| 4 | report_drafter | 보고서 초안 작성 | Milvus |
| 4 | peer_review | 가상 논문 리뷰 | Milvus |

### 5.3 에이전트 공통 패턴 (LangGraph StateGraph)

모든 에이전트는 동일한 패턴으로 구현:

```python
# 1. StateGraph 정의
graph = StateGraph(AgentState)

# 2. 노드 추가 (각 에이전트별 2~3개 노드)
graph.add_node("retrieve", retrieve_node)     # 검색
graph.add_node("generate", generate_node)     # LLM 답변 생성

# 3. 엣지 연결
graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)

# 4. 컴파일
agent = graph.compile()
```

## 6. 검색 전략

### 6.1 Hybrid Search (Milvus)

```
사용자 쿼리
  │
  ├── embed_query() → Dense 벡터 (1024d)
  │       │
  │       ▼
  │   AnnSearchRequest(dense, metric=IP, top_k)
  │
  └── query_text → Sparse (BM25)
          │
          ▼
      AnnSearchRequest(sparse, top_k)
  │
  ▼
RRFRanker (Reciprocal Rank Fusion)
  │
  ▼
상위 K개 결과 반환
```

### 6.2 원문 우선 검색 (paper_qa 전략)

```
1. DOI가 있으면 → MariaDB에서 원문 직접 조회
2. 제목이 있으면 → MariaDB에서 제목 검색
3. Fallback → Milvus Hybrid Search (chunk 단위)
```

### 6.3 청킹 전략 (`app/services/chunker_v2.py`)

기존 고정 크기 청킹 외에 5가지 고급 전략을 제공 (상세: `06_고급_청킹_설계서.md`):

| 전략 | 설명 | 비용 | 논문 적합도 |
|------|------|------|-----------|
| `fixed` | 고정 크기 문자 분할 (기존) | 없음 | ★★ |
| `recursive` | 구분자 우선순위 재귀 분할 (★기본값) | 없음 | ★★★★ |
| `sentence` | 문장 단위 분할 후 병합 | 없음 | ★★★ |
| `semantic` | 임베딩 유사도 기반 의미 경계 | 임베딩 | ★★★★ |
| `section` | 논문 섹션 구조 인식 + recursive (★★추천) | 없음 | ★★★★★ |

### 6.4 Multi-Query Retrieve (literature_survey 등)

```
1. LLM이 주제를 4-6개 섹션으로 분해
2. 각 섹션별 2-3개 검색 쿼리 생성
3. 각 쿼리로 Milvus 검색 (top_k_per_query=3)
4. 결과 병합 + 중복 제거
```

## 7. 날짜 파싱 시스템

### 7.1 4가지 패턴

| 패턴 | 예시 | 변환 결과 |
|------|------|-----------|
| D1: 절대 특정 | "2024년 11월" | 20241101~20241130 |
| D1: 절대 범위 | "2024년 10월~12월" | 20241001~20241231 |
| D2: 분기/반기 | "2023년 3분기" | 20230701~20230930 |
| D2: 연도 범위 | "2022~2024년" | 20220101~20241231 |
| D3: 상대 | "최근 6개월" | (현재-6M)~현재 |
| D3: 상대 | "작년 여름" | 서버시간 기준 계산 |
| D4: 비교 | "2020년과 2024년 비교" | 20200101~20241231 |

### 7.2 구현 방식
- **regex 기반 빠른 파싱** — 성능 우선
- **서버 시간 기준** — `get_current_datetime()`으로 현재 시간 확보
- **LLM 보완** — regex 실패 시 LLM이 날짜 추출

## 8. 스트리밍 아키텍처

```
POST /api/chat (stream=true)
  │
  ▼
[비스트리밍 구간]
  supervisor.ainvoke(state)
    - extract_dates, classify_intent, route_to_agent 실행
    - stream_mode=True → LLM 최종 호출 스킵
    - messages를 state.metadata._llm_messages에 저장
  │
  ▼
[SSE 스트리밍 구간]
  llm.chat_completion_stream(messages)
    ├── event: "token" → 실시간 토큰 전송
    ├── event: "sources" → 참조 문헌
    ├── event: "usage" → 토큰 사용량 (최종 생성 호출만)
    └── event: "done" → 스트리밍 종료

[토큰 usage 정책]
  - API 응답: 최종 답변 생성 LLM 호출의 usage만 반환 (항상 반환, 0이라도)
  - 내부 LLM 호출 (의도 분류, 조건 추출 등): Langfuse에서 개별 추적

[Langfuse]
  @observe로 감싸서 1 질문 = 1 trace 보장
  trace 안에서 모든 내부 LLM 호출이 개별 generation으로 확인 가능
```
