# 에이전트별 TTFT(첫 토큰 발행 시간) 분석

> 분석일: 2026-04-02

## 1. 개요

사용자 질문 입력 후 첫 토큰이 발행되기까지의 시간(TTFT)을 에이전트별로 분석한 문서입니다.
TTFT는 **Supervisor 공통 전처리** + **에이전트 내부 처리** 시간의 합입니다.

---

## 2. Supervisor 공통 전처리 (모든 에이전트 공통)

모든 질문은 Supervisor의 7단계 전처리를 거친 후 에이전트로 라우팅됩니다.

| 단계 | 함수 | LLM 호출 | DB 호출 | 시간 영향 |
|------|------|---------|--------|----------|
| 1. build_history | 대화 히스토리 구성 | 0 | 0 | 낮음 (문자열 처리) |
| 2. extract_dates | 날짜 추출 (Regex) | 0 | 0 | 낮음 (Regex) |
| 3. extract_conditions | 조건 추출 | **1회** | 0 | **높음** (LLM 대기) |
| 4. expand_domain_terms | 도메인 용어 확장 | 0 | 0 | 매우 낮음 (glossary lookup) |
| 5. classify_intent | 의도 분류 | **1회** | 0 | **높음** (LLM 대기) |
| 6. route_to_agent | 에이전트 라우팅 | 0 | 0 | - |
| 7. append_citation | 참조 문헌 추가 | 0 | 0 | 매우 낮음 (후처리) |
| **합계** | | **2회** | **0** | **LLM 2회가 고정 병목** |

## 4. 에이전트별 상세 처리 흐름

### 4.1 analytics (가장 빠름)

```
classify_analytics_type (LLM 1회)
  → fetch_data (MariaDB SQL 1~2회)
  → generate_response (최종 LLM)
```

- Milvus 검색 없음, SQL만 사용
- 예상 TTFT: **~2~2.5초** (Supervisor 포함)

### 4.2 paper_qa / paper_deep_dive

```
retrieve (경로 분기):
  A. DOI 명시 → MariaDB 1회 (가장 빠름)
  B. 제목 명시 → LLM 제목추출 1회 + MariaDB 1회
  C. 히스토리 참조 → 규칙 기반 추출 + MariaDB 1회
  D. Fallback → Milvus 임베딩+검색 1회
  → generate (최종 LLM)
```

- DOI/제목이 있으면 DB만 → 빠름
- Fallback시 Milvus 1회 → 중간
- 예상 TTFT: **~2~3초**

### 4.3 trend_analyzer / patent_landscaper / report_drafter / peer_review / knowledge_connector

```
retrieve_by_query (임베딩 1회 + Milvus 검색 1회)
  → generate/analyze (최종 LLM)
```

- 단일 Milvus 검색만 수행
- 예상 TTFT: **~2~3초**

### 4.4 literature_survey (가장 느림)

```
plan_survey (LLM 1회) → 4~6개 섹션 + 섹션당 2~3개 검색 쿼리 생성
  → multi_retrieve:
    섹션1: embed(q1) → search(q1) → embed(q2) → search(q2)
    섹션2: embed(q3) → search(q3) → embed(q4) → search(q4)
    ...
    (총 8~18회 순차 실행)
  → synthesize (최종 LLM)
```

- 예상 TTFT: **~7~14초** (검색 횟수에 비례)

### 4.5 idea_generator

```
decompose_topic (LLM 1회) → 3~5개 sub_topic + topic당 2~3개 쿼리
  → cross_retrieve:
    sub_topic1: embed(q1) → search(q1) → embed(q2) → search(q2)
    sub_topic2: embed(q3) → search(q3) → ...
    (총 6~15회 순차 실행)
  → generate_ideas (최종 LLM)
```

- 예상 TTFT: **~5~10초**

### 4.6 cross_domain

```
extract_problem (LLM 1회) → 3~4개 타 도메인 + 도메인당 2개 쿼리
  → search_analogies:
    도메인1: embed(q1) → search(q1) → embed(q2) → search(q2)
    도메인2: ...
    + display 원본 검색 1회
    (총 7~9회 순차 실행)
  → map_to_display (최종 LLM)
```

- 예상 TTFT: **~5~8초**

### 4.7 competitive_intel

```
search_competitor_news:
  경쟁사1: embed(query) → search → ...
  경쟁사5: embed(query) → search
  + 원본 쿼리 1회
  (총 6회 순차 실행)
  → generate_briefing (최종 LLM)
```

- 예상 TTFT: **~4~6초**

### 4.8 experiment_planner / material_advisor

```
parse_hypothesis/goal (LLM 1회) → 3개 검색 쿼리
  → search_methods/materials:
    embed(q1) → search(q1) → embed(q2) → search(q2) → embed(q3) → search(q3)
    (총 3회 순차 실행)
  → design/compare (최종 LLM)
```

- 예상 TTFT: **~3~5초**

## 5. TTFT 구성 요소별 소요 시간 (추정)

| 구성 요소 | 단일 호출 소요 시간 (추정) | 비고 |
|----------|----------------------|------|
| LLM 호출 (llm_json_call) | 300~500ms | 짧은 JSON 출력 |
| LLM 호출 (최종 답변 첫 토큰) | 300~500ms | 스트리밍 시작까지 |
| 임베딩 생성 (embed_query) | 200~400ms | 네트워크 I/O 포함 |
| Milvus hybrid_search | 200~500ms | dense + sparse + RRF |
| MariaDB SQL 쿼리 | 50~200ms | 인덱스 활용 시 |
| 문자열/Regex 처리 | <10ms | CPU 작업 |

## 7. 병목 원인 요약

### 공통 병목
- Supervisor의 LLM 2회 호출 (extract_conditions + classify_intent): **~0.6~1초 고정 지연**

### 느린 그룹 핵심 병목
- `multi_query_retrieve()` 내부에서 **임베딩 → Milvus 검색을 쿼리마다 순차 실행**
- 쿼리 N개일 때: `N × (임베딩 시간 + 검색 시간)` = `N × 0.5~0.9초`

```
현재 구조 (순차):
  embed(q1) → search(q1) → embed(q2) → search(q2) → ... → embed(qN) → search(qN)
  총 시간: N × (embed + search)
```

### 속도 차이가 나는 이유
- **빠른 그룹**: 검색 0~1회 → 고정 시간
- **느린 그룹**: 검색 3~18회 → 횟수에 비례하여 선형 증가

## 8. 향후 최적화 검토 포인트

> 아래는 구조 변경 없이 검토할 수 있는 방향입니다.

1. **임베딩 배치 처리**: 여러 쿼리를 한 번에 임베딩 API로 전송 (API가 batch 지원하는 경우)
2. **Milvus 검색 병렬화**: `asyncio.gather()`로 독립적인 검색을 동시 실행
3. **임베딩 캐시**: 동일/유사 쿼리 반복 시 캐싱
4. **Supervisor LLM 통합**: extract_conditions + classify_intent를 단일 호출로 합침
5. **동적 검색 횟수 조절**: 사용자 요청 복잡도에 따라 섹션/쿼리 수 조정

## 9. 참고: 스트리밍 모드

모든 에이전트는 스트리밍을 지원합니다. 스트리밍 모드에서는:
- 에이전트가 검색/context 준비를 완료한 후
- API 레이어에서 최종 LLM 호출을 스트리밍 처리
- TTFT 자체는 동일하나, 첫 토큰부터 사용자에게 표시되어 **체감 대기 시간 감소**
